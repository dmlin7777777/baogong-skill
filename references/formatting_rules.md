# Resume Formatting Rules

Universal formatting rules for the Writer node. Apply these during CP4 (Wording Upgrade) and CP5 (Draft Finalization). These are operational-level rules — concrete, immediately actionable, independent of JD or region.

## Contact Information

- **No QR codes** — takes space, unusable in print/PDF, and looks unprofessional in most regions
- **Consolidate phone + messaging**: `13500000000 (WeChat)` on one line, not separate entries
- **Email must be professional**: no `cutegirl123@qq.com`. If user has no professional email, flag it
- **LinkedIn/GitHub/Portfolio**: include only if profile is maintained and relevant. Dead links are worse than no links
- **Physical address**: full address is unnecessary in most regions. City + Country is sufficient

## Certifications & Education

- **Never list certifications still in progress** ("备考中: CPA") — studying ≠ holding. Exception: if already passed partial exams, write "CPA (passed X, Y subjects)"
- **Course grades: all or nothing** — listing only high scores (91, 93, 87) signals that other scores are worse. Either list all major course grades or none
- **Coursework vs tools**: programming languages and software belong in "Skills", not "Relevant Courses". Courses are academic subjects
- **Awards belong under Education**, not in a separate "Campus Activities" section — saves space and keeps context together

## Skills Section

- **Don't split basic suites**: "Office (advanced Excel: VLOOKUP, pivot tables)" not "Word, Excel, PowerPoint" listed separately
- **Group by function, not alphabetically**: Data (SQL, Python, R) / Visualization (Tableau, Power BI) / Languages (English, Mandarin)
- **Proficiency markers must be honest**: "Proficient" means you can be tested on it in an interview. "Familiar" means you've used it but wouldn't want to be quizzed
- **Don't list skills with no project evidence** — if the skill doesn't appear in any experience bullet, it's a red flag for interviewers

## Bullet Writing

- **Bold summary prefix must be short** (2-4 words) and hit JD keywords. No adjectives in the prefix
  - ❌ "Efficiently managed comprehensive data analysis workflows"
  - ✅ "**Data Pipeline**: Built automated ETL..."
- **No vague quantifiers**: "many", "multiple", "various", "significant", "extensive" are banned. Replace with numbers or remove
  - ❌ "Assisted multiple clients" → ✅ "Assisted 30+ clients"
  - ❌ "Processed large volumes of data" → ✅ "Processed 2M+ records daily"
  - If the number is genuinely unknown, mark `[confirm: how many?]` during drafting — never guess
- **One achievement per bullet** — don't chain two unrelated outcomes with "and". Split into separate bullets
- **No orphan bullets** — if a role has only 1 bullet, either expand it or merge the role into another entry. A single bullet signals thin experience

## Formatting Consistency

- **Date format**: pick one and stick to it throughout. `YYYY.MM` or `MMM YYYY` — never mix
- **Tense**: past tense for completed roles, present tense for current role only
- **Parallel structure**: all bullets in one section should start with the same part of speech (verb-first preferred)
- **Page target**: 1 page for <10 years experience, 2 pages max for senior roles. Never exceed 2 pages

## What NOT to Fix

These rules are for the Writer node during drafting. Do NOT apply them retroactively to the user's source resume during Init-A (Master Resume creation) — Init-A captures facts as-is without editorial changes.
